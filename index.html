Overview

The core of a full-stack payment ecosystem

The Payment Gateway SDK is the technical foundation of a fintech platform offering payment infrastructure to merchants across Nigeria. Rather than being a standalone product, the SDK is the shared payment layer embedded across multiple distinct products — from a merchant-facing storefront to a backoffice used by platform administrators.

It serves as a unified integration layer that enables merchants to securely accept payments across multiple channels without needing to manage separate provider integrations or complex payment flows.

This case study explores how the SDK was designed to be embeddable, extensible, and production-grade — supporting diverse payment methods while maintaining a consistent developer and user experience across every surface it powers.

The Challenge

One payment experience, many surfaces

Fintech platforms in the Nigerian market face a compounded problem: payment methods are fragmented across providers, each with its own integration contract. Building that integration once per product is expensive — building it five times is unsustainable.

The challenge was to design a single SDK that could be embedded across radically different contexts — a merchant storefront, a public-facing website, a standalone payment page, and a pension checkout — without requiring each product team to reason about the underlying payment rails.

Fragmented payment rails

Card, transfer, QR code, OPay wallet, Google Pay, and Apple Pay each have distinct authentication and completion flows that needed unification.

Multi-product embedding

Five separate Angular and Next.js products all needed payment capability without duplicating integration logic across codebases.

Merchant-specific checkout

Each merchant's branding, fees, and available methods differ. The SDK had to be configurable at runtime without rebuild.

Frontend–backend synchronization

Ensuring accurate transaction status even when users leave the page, complete payments externally, or experience network interruptions.

Regulatory and security constraints

PCI-DSS compliance expectations, OTP flows, and third-party provider tokens had to be handled consistently and securely.

Solution

Payment rails as a single, composable layer

The SDK exposes a clean, product-agnostic API that abstracts all provider-specific logic. A consuming product initializes the SDK with a transaction context — amount, merchant config, available methods — and the SDK handles the rest:

  • rendering the payment UI
  • routing to the correct provider
  • managing polling or webhook callbacks
    • and
  • emitting a result event.

"Embed once, pay any way — the SDK handles routing, UI, and provider negotiation invisibly."

Architecture

How it's built

The SDK is structured as a self-contained module with a well-defined initialization interface. Its internal architecture separates concerns cleanly: a routing layer determines which payment rail is active, provider adapters normalize external APIs, and a shared state machine drives the UI through transaction states.

Outcomes

What this work delivered

Single source of truth for payments

All payment logic lives in one place. Bug fixes, new payment methods, and compliance updates propagate to every product automatically.

Consistent checkout UX across surfaces

Customers get the same familiar payment experience whether checking out from the storefront, a payment page, or the website.

Accelerated product iteration

New products could integrate payments in hours, not weeks. The storefront and payment pages were shipped faster because checkout was already solved.

Merchant-configurable without redeployment

Available payment methods, fees, and branding are injected at runtime via merchant config — no rebuild needed per merchant or campaign.

Reflection

What this project demonstrates

More than any single feature, this project demonstrates the ability to think at the platform level — not just building what's needed today, but designing shared infrastructure that scales across an entire product family. The SDK represents a deliberate architectural choice: accept complexity in one place so every consumer can stay simple.

Working across five products also required deep fluency in API design, state management, and asynchronous payment flows — skills that are transferable to any product that involves real money, real time, and real users at scale.

What I learned

Designing payment systems goes beyond UI — it requires thinking in terms of reliability, timing, and user trust.

Handling asynchronous flows is essential in real-world fintech systems.

Developer experience directly impacts adoption and scalability.

What I would improve

Introduce webhook support alongside polling for more efficient updates

Improve SDK documentation and onboarding for external developers

Add analytics hooks for better merchant insights

Strengthen retry and edge-case error handling

"The best infrastructure is invisible to the teams who build on it — and memorable to the users who experience it."
